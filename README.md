# 外卖点餐系统

## 项目简介

该项目是一个基于 **Spring Boot + MyBatis-Plus + MySQL + Redis** 实现的外卖点餐系统，项目包含后台管理端和移动端用户端两部分，覆盖了餐饮外卖业务中常见的员工管理、菜品管理、套餐管理、用户登录、地址管理、购物车和下单等核心流程。

本项目主要用于模拟真实餐饮门店的线上点餐场景，解决传统餐饮业务中菜品维护分散、用户点餐流程不统一、订单数据难以管理、后台与用户端数据不同步等问题。通过该项目可以完整理解 Java Web 项目从接口开发、数据库操作、权限校验、缓存优化到业务流程落地的实现过程。

---

## 项目解决的问题

### 1. 餐饮门店菜品管理不统一

传统线下餐饮门店通常依赖人工维护菜单，菜品分类、菜品价格、菜品状态和套餐信息容易出现管理混乱的问题。本项目通过后台管理系统实现了菜品、分类、套餐的统一维护。

实现效果：

- 支持菜品分类管理
- 支持菜品新增、修改、删除、分页查询
- 支持菜品口味管理
- 支持套餐管理和套餐菜品关联
- 支持菜品、套餐启售和停售

---

### 2. 用户点餐流程不完整

用户端需要完成从登录、浏览菜品、加入购物车、选择地址到提交订单的完整链路。本项目实现了较完整的移动端点餐流程。

实现效果：

- 用户通过手机号验证码登录
- 登录后可以浏览菜品和套餐
- 支持菜品加入购物车
- 支持查看购物车和清空购物车
- 支持选择地址并提交订单
- 订单提交后生成订单主表和订单明细数据

---

### 3. 后台接口缺少权限控制

如果后台接口没有统一权限校验，未登录用户也可能访问管理接口，存在安全风险。本项目使用登录过滤器对请求进行统一拦截，区分后台员工和移动端用户登录状态。

实现效果：

- 后台员工登录后才能访问管理接口
- 移动端用户登录后才能访问用户相关接口
- 静态资源、登录接口、短信验证码接口允许放行
- 未登录访问时返回 `NOTLOGIN`
- 使用 `ThreadLocal` 保存当前登录用户 ID，方便业务层获取当前用户信息

---

### 4. 热点数据重复查询数据库

菜品列表、套餐列表属于高频访问数据，如果每次都直接查询数据库，会增加数据库压力。本项目引入 Redis 和 Spring Cache，对热点数据进行缓存。

实现效果：

- 菜品列表使用 Redis 缓存
- 套餐列表使用 Spring Cache 缓存
- 修改菜品或套餐时清理相关缓存
- 降低高频查询对 MySQL 的访问压力
- 提升用户端菜品展示速度

---

### 5. 下单业务需要保证数据一致性

订单提交涉及购物车、用户信息、地址信息、订单主表、订单明细表等多张表，如果处理不当容易出现订单数据不完整的问题。本项目在订单提交逻辑中使用事务进行控制。

实现效果：

- 根据当前用户查询购物车数据
- 校验购物车是否为空
- 校验用户地址是否有效
- 自动生成订单号
- 计算订单总金额
- 保存订单主表数据
- 批量保存订单明细数据
- 使用事务保证订单数据一致性

## 核心实现说明

### 1. 统一返回结果封装

项目使用 `R<T>` 对接口返回结果进行统一封装，前端可以通过固定格式判断请求是否成功，减少不同接口返回格式不一致的问题。

主要解决：

- 接口返回结构不统一
- 前端处理响应数据复杂
- 错误信息无法统一维护

---

### 2. 全局异常处理

项目通过全局异常处理器统一捕获业务异常和数据库异常，避免异常信息直接暴露给前端。

主要解决：

- 接口异常返回不友好
- 重复编写 `try-catch`
- 业务异常无法统一提示

---

### 3. 登录校验过滤器

项目使用 `LoginCheckFilter` 对请求进行统一拦截。过滤器会判断当前请求是否属于放行路径，如果不是，则检查 Session 中是否存在后台员工或移动端用户登录信息。

主要解决：

- 未登录用户访问受保护接口
- 后台管理接口缺少权限校验
- 用户端接口和后台接口登录状态混乱

---

### 4. ThreadLocal 保存当前用户

在登录过滤器中获取当前登录用户 ID，并保存到 `BaseContext` 中。业务代码可以直接获取当前用户 ID，用于购物车、地址、订单等与用户强相关的操作。

主要解决：

- 多个业务方法重复传递用户 ID
- 用户数据归属不清晰
- 多线程环境下用户信息隔离问题

---

### 5. MyBatis-Plus 简化数据库操作

项目大量使用 MyBatis-Plus 提供的 `BaseMapper`、`IService`、`LambdaQueryWrapper` 和分页插件完成数据操作。

主要解决：

- 重复编写基础 CRUD SQL
- 查询条件拼接复杂
- 分页查询实现繁琐

---

### 6. Redis 缓存热点数据

项目对菜品列表、套餐列表、短信验证码等数据使用 Redis 缓存。菜品列表通过 `RedisTemplate` 手动维护缓存，套餐列表通过 Spring Cache 注解实现缓存。

主要解决：

- 用户端频繁查询菜品导致数据库压力大
- 验证码不适合长期存储在数据库中
- 热点数据响应速度慢

---

### 7. 订单提交事务处理

订单提交涉及订单表和订单明细表，需要保证数据同时成功或同时失败。项目在订单提交方法中使用 `@Transactional` 保证事务一致性。

主要解决：

- 订单主表保存成功但明细保存失败
- 购物车为空仍然下单
- 地址信息错误导致订单异常
- 订单金额计算和明细数据不一致

## 业务流程

### 用户下单流程

```text
用户登录
   ↓
浏览菜品 / 套餐
   ↓
加入购物车
   ↓

   ↓

   ↓
计算订单金额
   ↓
生成订单主表
   ↓

   ↓
返回下单成功
```

## 快速启动

### 1. 克隆项目

```bash
git clone https://github.com/yeyang520/reggie_take_out.git
cd reggie_take_out
```

### 2. 创建数据库

在 MySQL 中创建数据库：

```sql
CREATE DATABASE reggie DEFAULT CHARACTER SET utf8mb4 COLLATE utf8mb4_general_ci;
```

然后导入项目所需的数据表 SQL 文件。

> 注意：如果仓库中没有提供 SQL 文件，需要根据实体类自行创建表，或补充瑞吉外卖项目对应的数据库脚本。

### 3. 修改配置文件

修改 `src/main/resources/application.yml` 中的数据库和 Redis 配置：

```yaml
server:
  port: 8080

spring:
  datasource:
    druid:
      driver-class-name: com.mysql.cj.jdbc.Driver
      url: jdbc:mysql://localhost:3306/reggie?serverTimezone=Asia/Shanghai&useUnicode=true&characterEncoding=utf-8&useSSL=false&allowPublicKeyRetrieval=true
      username: root
      password: 123456
  redis:
    host: 127.0.0.1
    port: 6379
    database: 0
```

同时可以根据本地环境修改图片上传路径：

```yaml
reggie:
  path: D:\img\
```

### 4. 启动 Redis

确保 Redis 服务已启动：

```bash
redis-server
```

### 5. 启动项目

使用 Maven 启动：

```bash
mvn spring-boot:run
```

或者直接运行：

```text
src/main/java/boy/learning/reggie/ReggieApplication.java
```

### 6. 访问系统

后台管理端：

```text
http://localhost:8080/backend/index.html
```

移动端用户端：

```text
http://localhost:8080/front/index.html
```
